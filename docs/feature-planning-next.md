# cursor-consumption 次期機能企画

## 前提

- Primary ユーザーは Cursor Pro/Pro+/Ultra の個人開発者、Secondary ユーザーはチーム管理者。
- 現在のプロダクトは Next.js App Router、Prisma、PostgreSQL、`pricing/models.json`、Cursor Hooks 経由のイベント収集を前提にする。
- 既存実装では Hook イベントの保存とイベント種別別ダッシュボードが先行している。次の価値は、保存済みイベントを「利用状況の理解」と「次の行動」に変換すること。
- 非公式 API や Cursor 内部仕様への依存は最小化し、取得できたデータの確度を明示する。

## 次に企画すべき機能候補 3つ

| 候補 | 機能名 | 解決する課題 | 対象ユーザー | MVP/Post-MVP/将来 | 優先度 |
|---|---|---|---|---|---|
| 1 | Hook由来のトークン・コスト概要ダッシュボード | 「今日・今月いくら使っているか」「どのモデル/モードが高いか」が分からず、使い方を変える判断ができない | 個人開発者 | MVP | 最優先 |
| 2 | データ信頼性・プライバシー診断 | Hooks が止まった、DB保存に失敗した、プロンプト等のセンシティブ情報が保存される、という不安で継続利用しづらい | 個人開発者、チーム管理者 | MVP後半/Post-MVP | 高 |
| 3 | 月末予測と予算アラート | 月末に上限超過や追加課金に気づき、利用ペースを調整できない | 個人開発者、チーム管理者 | Post-MVP | 中 |

### 候補 1: Hook由来のトークン・コスト概要ダッシュボード

- **概要**: `AgentEvent` 等に保存される token usage とモデル情報を集計し、当日/当月の推定コスト、モデル別内訳、Composer/Agent/Ask 等の利用傾向を表示する。
- **提供価値**: ユーザーが「どこで消費しているか」を短時間で理解し、高コストモデル利用、長時間Agent実行、コンテキスト肥大化などの見直しに進める。
- **選定理由**: 既存のHook保存・料金テーブル・コスト計算方針を最も直接的にユーザー価値へ変換できる。

### 候補 2: データ信頼性・プライバシー診断

- **概要**: Hook受信状況、直近同期時刻、イベント欠損の可能性、保存されるフィールドの種類、ローカル保存範囲を確認できる診断画面を提供する。
- **提供価値**: 個人開発者がローカルで安心して使える。チーム導入時も「何が保存されるか」を説明しやすい。
- **選定理由**: 非公式・ローカル収集ツールの継続利用において信頼性と透明性が重要なため。

### 候補 3: 月末予測と予算アラート

- **概要**: 現在の消費ペースから月末推定コストを算出し、ユーザー設定の閾値を超える見込みならUI上で警告する。
- **提供価値**: 使いすぎに気づき、モデル選択やAgent利用頻度を調整できる。
- **選定理由**: 既存要件でもPost-MVPとして明示されており価値は高いが、正確な日次コストの可視化が先に必要。

# PRD: Hook由来のトークン・コスト概要ダッシュボード

## 1. ユーザー課題

### Primary: Cursor Pro/Pro+/Ultra の個人開発者

- **場面**: 日常的に Cursor Agent、Composer、Ask、Tab補完を使いながら個人開発を進めている。
- **困りごと**:
  - Cursor の標準表示だけでは、今日/今月の利用ペースとコスト感をすぐ把握しづらい。
  - どのモデル、どのモード、どの作業セッションで消費が増えたのか分からない。
  - 「モデルを変える」「Agentに投げる粒度を変える」「コンテキストを整理する」など、次の行動に結びつけにくい。
  - 非公式APIや内部仕様に依存する数値の場合、正確性の限界が分からず信用しきれない。

### Secondary: チーム管理者

- **場面**: チーム内で Cursor 利用が増え、将来的な予算管理や利用傾向把握をしたい。
- **困りごと**:
  - Enterprise/Admin API 連携前でも、ローカルまたは小規模チームで概算傾向を確認したい。
  - メンバーに「何が高コストになりやすいか」を説明する材料が不足している。

## 2. 企画する機能

### 機能名

Hook由来のトークン・コスト概要ダッシュボード

### 概要

Cursor Hooks で保存されたイベントから、トークン利用量と推定コストを集計して表示する。まずは個人開発者が毎日確認するトップ画面として、当日・当月の推定コスト、モデル別内訳、モード別内訳、直近の高消費イベントを提供する。

### 対象ユーザー

- Primary: Cursor Pro/Pro+/Ultra の個人開発者
- Secondary: 小規模チームの管理者、将来のBusiness/Enterprise連携利用者

### 提供価値

- 自分のCursor利用が「どの程度のコスト感か」を即座に理解できる。
- 高コストの原因をモデル、モード、セッション単位で特定し、次の行動を選びやすくする。
- 数値が推定か実額か、データが十分か不足しているかを明示し、ツールへの信頼性を高める。

## 3. ユースケース

### ユースケース A: 今日の使いすぎに気づく

個人開発者が作業終盤にダッシュボードを開く。当日推定コストが普段より高く、モデル別内訳では高単価モデルのAgent利用が大半を占めている。ユーザーは以降の軽い相談を低コストモデルやAskに切り替える。

### ユースケース B: コンテキスト肥大化を把握する

ユーザーが長時間のAgent作業後に直近高消費イベントを見る。`preCompact` のコンテキスト使用率や `afterAgentResponse` のトークン数が増えていることに気づき、会話を分ける、不要なファイル参照を減らすなどの対応を行う。

### ユースケース C: 月次の使い方を振り返る

月末にユーザーが当月サマリーを確認する。特定プロジェクトのAgent利用が多く、Composer/Agentの比率が高いことが分かる。翌月は高コスト作業を分割し、手元実装とAI依頼の境界を見直す。

### ユースケース D: チーム導入前の説明材料にする

チーム管理者が自分のローカル利用データをもとに、モデル別・モード別の消費傾向をチームに共有する。公式Admin API連携前の仮説作りとして使う。

## 4. MVPスコープ

### 最初に作るもの

1. **コスト概要トップ**
   - 今日の推定コスト
   - 今月の推定コスト
   - 今日の入力/出力/キャッシュ読み取り/キャッシュ書き込みトークン
   - 直近データ受信時刻
   - データ確度ラベル: `estimated`, `partial`, `no-data`

2. **モデル別内訳**
   - モデル別の推定コスト
   - モデル別の総トークン
   - モデル名不明イベントは `unknown model` として表示

3. **モード/イベント別内訳**
   - Agent、Session、Tool、File など既存Hookイベント単位の消費傾向
   - 可能な場合は `composerMode`、`hookEventName` で分類

4. **直近の高消費イベント**
   - 発生時刻
   - モデル
   - イベント種別
   - 入力/出力/キャッシュ系トークン
   - 推定コスト
   - transcript path や prompt 本文は初期表示しない

5. **透明性表示**
   - 「Cursor請求額そのものではなく、Hookイベントと料金テーブルからの推定値」であることを画面に明記
   - 使用した料金テーブルのバージョンまたは更新日を表示
   - token usage が取得できないイベントは集計対象外であることを明記

### MVPで作らないもの

- デスクトップ通知、Slack通知、メール通知
- 自動予算アラート
- Cursor内部APIからの請求額同期
- Enterprise/Admin API連携
- 複数ユーザーの権限管理
- CSV/JSONエクスポート
- プロンプト本文の分析やLLMによる改善提案
- 正確なCursor請求額との自動照合

### Post-MVP

- 月末予測と予算閾値
- 日次/週次/月次トレンド
- プロジェクト/ワークスペース別内訳
- データ信頼性診断との統合
- CSV/JSONエクスポート

### 将来拡張

- Cursor公式APIまたはAdmin APIが利用可能な場合の実額連携
- チームメンバー別、Billing Group別の集計
- 料金テーブルの自動更新と差分通知
- 利用パターンに応じたコスト最適化提案

## 5. 成功指標

### 定量指標

- 初回データ保存後、トップ画面で当日推定コストが表示されるユーザー比率: 80%以上
- ダッシュボード表示時の主要集計APIレスポンス: 500ms以内
- モデル別/イベント別内訳の `unknown` 比率: 30%以下を目標に改善
- 直近7日以内にダッシュボードを2回以上開いたユーザー比率
- データ確度ラベルが `no-data` から `estimated` または `partial` に遷移したユーザー比率

### 定性指標

- ユーザーが「何にコストがかかっているか」を説明できる。
- ユーザーが「次に何を調整すべきか」を1つ以上挙げられる。
- 推定値の限界が分かり、誤って請求額そのものと理解しない。
- ローカル保存・プロンプト非表示方針に安心感がある。

## 6. 優先度

### 評価

| 観点 | 評価 | 理由 |
|---|---|---|
| Impact | 高 | コスト可視化というプロダクトの中核価値に直結し、個人開発者の毎日の意思決定に使える |
| Confidence | 中 | Hook経由のtoken usageは既存スキーマにあるが、全イベントで取得できる保証はなく推定値として扱う必要がある |
| Effort | 中 | 既存のPostgreSQL/Prisma/Next.js基盤を使える一方、集計クエリ、UI、データ確度表示が必要 |

### 優先度判断

最優先で実施する。理由は、既存のHook保存機能だけでは「記録できている」価値に留まり、ユーザーのコスト理解や行動変容につながりにくいため。まず推定値でもダッシュボード上で利用傾向を見える化し、その後に予測・アラート・公式API連携へ進む。

## 7. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| Cursor Hooks の仕様変更 | token usage やイベント名が変わり、集計できなくなる | 既知フィールドのみを集計し、未知/欠損は `partial` として表示。Hookイベント保存は後方互換を重視する |
| 非公式API利用のリスク | 利用規約・仕様変更・認証変更の影響を受ける | MVPでは内部API請求額取得に依存しない。Hooks由来データとローカル料金テーブルによる推定に限定する |
| データ精度の誤解 | 推定値を請求額そのものと誤認し、判断を誤る | 画面上で「推定」「部分データ」「料金テーブル更新日」を明示する。実額との一致を保証しない |
| 料金体系変更 | `pricing/models.json` が古いとコスト推定がずれる | 料金テーブルのバージョン/更新日を表示し、未知モデルや古い料金表を警告する |
| セキュリティ/プライバシー | プロンプト、ファイルパス、ユーザーメール等が不用意に露出する | 概要画面ではprompt本文やfile contentを表示しない。詳細リンクも初期は最小限にし、必要時のみ表示 |
| ローカルDB未起動/データなし | 初回体験で空画面になり価値が伝わらない | `no-data` 状態でセットアップ確認、Hook受信状況、次に試す操作を案内する |
| パフォーマンス | イベント件数増加でトップ画面が重くなる | 日付範囲を限定し、必要に応じてDailySummaryや集計ビューを利用する |
| UXの複雑化 | 数値が多すぎて次の行動が分からない | MVPは概要、内訳、高消費イベントに絞る。最適化提案はPost-MVPに回す |
| チーム利用への拡張性 | 個人前提の設計が後のチーム集計を阻害する | 集計軸に userEmail/workspaceRoot を残し、UI上は個人向けに限定する |

## 8. 開発チームへの要求

### 画面

1. **`/` ダッシュボードトップの拡張**
   - 現在のHook Events Dashboardを、コスト概要を主役にしたトップへ更新する。
   - 既存のイベント種別カードは「収集状況」セクションとして残す。
   - 表示項目:
     - 今日の推定コスト
     - 今月の推定コスト
     - 今日の総トークン
     - 直近イベント受信時刻
     - データ確度ラベル
     - モデル別コスト上位
     - イベント/モード別コスト上位
     - 直近高消費イベント

2. **空状態/部分状態**
   - データなし: HookセットアップとPostgreSQL起動確認を案内する。
   - token usageなし: 「イベントは受信できているが、トークン情報がないためコスト推定できない」と表示する。
   - 未知モデル: 料金テーブル未対応の可能性を表示する。

### API

1. **`GET /api/usage/summary`**
   - 入力: `range=today|month` または `from`/`to`
   - 出力:
     - `estimatedCostCents`
     - `inputTokens`
     - `outputTokens`
     - `cacheReadTokens`
     - `cacheWriteTokens`
     - `eventCount`
     - `pricedEventCount`
     - `lastEventAt`
     - `dataQuality`

2. **`GET /api/usage/by-model`**
   - 入力: `range` または `from`/`to`
   - 出力: モデル別のトークン、推定コスト、イベント数、価格テーブル一致有無

3. **`GET /api/usage/by-event`**
   - 入力: `range` または `from`/`to`
   - 出力: `hookEventName`、`composerMode` などの分類別集計

4. **`GET /api/usage/high-consumption-events`**
   - 入力: `range`、`limit`
   - 出力: 高消費順のイベント一覧。prompt本文やfile contentは返さない。

### データ

- 集計対象の初期候補:
  - `AgentEvent.inputTokens`
  - `AgentEvent.outputTokens`
  - `AgentEvent.cacheReadTokens`
  - `AgentEvent.cacheWriteTokens`
  - `AgentEvent.model`
  - `AgentEvent.hookEventName`
  - `SessionEvent.composerMode`
  - `loggedAt`
- `UsageEvent` が既に使える場合は将来の正規化先として残す。ただしMVPではHook由来データを直接集計してもよい。
- 料金計算は既存の `pricing/models.json` と `CostCalculator` 方針を使う。
- `chargedCents` が存在するデータでは実額を優先し、なければ推定コストを使う。ただしHook由来MVPでは多くが推定になる前提。

### 状態

| 状態 | 条件 | UI表示 |
|---|---|---|
| `no-data` | Hookイベントが0件 | セットアップ案内、DB接続確認 |
| `events-only` | Hookイベントはあるがtoken usageがない | 収集は成功、コスト推定不可と説明 |
| `partial` | token usageはあるが未知モデルや欠損がある | 推定値と未計算件数を表示 |
| `estimated` | token usageと料金テーブルが概ね揃う | 推定コストとして表示 |
| `priced` | `chargedCents` 等の請求額に近い値がある | 将来状態。実額優先で表示 |

### 例外ケース

- `model` が null または空: `unknown model` として集計し、価格計算対象外にする。
- 料金テーブルにモデルがない: コストは未計算、トークンだけ集計する。
- token数が負数、NaN、極端に大きい: 集計対象外にし、データ品質に警告を出す。
- DB接続エラー: APIは500を返し、UIでは設定確認を案内する。
- 日付範囲が不正: APIは400を返す。
- タイムゾーン: MVPではローカル環境のタイムゾーンまたは設定値に基づく「今日」を使い、画面に基準を表示する。
- プロンプト/ファイル内容: 概要APIでは返却しない。

### 受け入れ条件

- Hookイベントに token usage がある環境で、トップ画面に今日/今月の推定コストが表示される。
- 料金テーブルにないモデルが含まれても画面が壊れず、未計算として表示される。
- データがない初回状態で、ユーザーが次に確認すべきセットアップ項目を理解できる。
- prompt本文やfile contentが概要APIレスポンスに含まれない。
- 集計APIのテストで、既知モデル、未知モデル、token欠損、空データのケースが確認されている。
