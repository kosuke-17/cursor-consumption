# cursor-consumption 次期機能企画

## 現在のプロダクト状況

`cursor-consumption` は、Cursor Hooks から発火するイベントを Next.js Route Handler 経由で PostgreSQL に保存し、Hook Events Dashboard でカテゴリ別に確認できる段階にある。既存設計では `UsageEvent`、`DailySummary`、料金テーブル外部化、コスト計算ロジックも用意されており、Cursor のAI利用量・トークン消費・コスト可視化へ拡張できる土台がある。

一方で、Primaryユーザーである Cursor Pro/Pro+/Ultra の個人開発者にとって重要な「今月どれだけ使ったか」「月末に超過しそうか」「どのモデル・機能・作業が高コストか」「次に何を変えるべきか」は、まだ一目で判断しづらい。次期企画では、Hooks経由のデータ収集を活かしつつ、非公式API依存を急に増やさず、ローカルで安心して使えるコスト可視化体験を優先する。

---

## 次に企画すべき機能候補 3つ

### 候補1: 月次コスト・予算ダッシュボード

#### 1. ユーザー課題

- Cursor Pro/Pro+/Ultra の個人開発者は、月の途中で「今のペースだとプラン内に収まるか」を把握しづらい。
- Cursor標準の表示は簡易的で、作業単位・モデル単位の要因分析までつながりにくい。
- 従量課金化により、月末に想定外の追加コストに気づくリスクがある。

#### 2. 企画する機能

- **機能名**: 月次コスト・予算ダッシュボード
- **概要**: 当月の推定消費額、プラン上限、残予算、月末着地予測、主要なコスト要因をトップページで表示する。
- **対象ユーザー**: Primary: 個人開発者 / Secondary: 小規模チーム管理者
- **提供価値**: 「今月このままで大丈夫か」と「何を見直せばよいか」を短時間で判断できる。

#### 3. ユースケース

個人開発者が朝にダッシュボードを開き、今月の消費がプランの70%に達していることと、月末予測がプランを超えることを確認する。高コスト要因が Agent + 高価なモデルに偏っていると分かり、重い調査タスクは必要時だけMax Modeにする、軽い質問は低コストモデルに寄せる、といった行動を取る。

#### 4. MVPスコープ

作るもの:
- 当月消費額、残予算、消費率、月末予測
- モデル別・機能別のコスト上位表示
- データ鮮度と推定精度ラベル
- プラン設定: Pro / Pro+ / Ultra / カスタム予算

作らないもの:
- デスクトップ通知
- チームメンバー別管理
- 経費レポート出力
- Cursor非公式APIへの新規依存拡大

#### 5. 成功指標

- 定量:
  - ダッシュボード初回表示から主要指標を理解できるまでの時間が短い
  - 月次ダッシュボードの継続閲覧率
  - 予算超過予測カードからモデル・機能内訳を見る遷移率
- 定性:
  - ユーザーが「今月の使いすぎ要因」を説明できる
  - ユーザーが「次に控える/変える利用」を判断できる

#### 6. 優先度

| 観点 | 評価 | 理由 |
|---|---|---|
| Impact | 高 | コスト可視化というプロダクト価値の中心に直結する |
| Confidence | 中 | Hooks由来データで概算可能だが、Cursor公式課金額との完全一致は保証しづらい |
| Effort | 中 | 既存のUsageEvent/DailySummary/CostCalculatorを活かせるが、集計APIと画面設計が必要 |

**総合優先度: P0**

#### 7. リスクと対策

- 仕様変更: Hook payload やモデル名が変わる可能性があるため、未知モデルは「未分類/推定不可」として表示する。
- データ精度: chargedCents がない場合は料金テーブル計算であることをUIに明示する。
- セキュリティ: プロンプト本文やトランスクリプト内容をコスト集計に使わない。
- UX: 推定値を断定的に見せず、「推定」「データ最終更新」を常に表示する。
- 運用: 料金テーブル更新日とバージョンを表示し、古い場合は警告する。

#### 8. 開発チームへの要求

- 画面: `/` を月次コストサマリー中心に再設計する。
- API: `GET /api/usage/summary`、`GET /api/usage/models`、`GET /api/usage/features` を用意する。
- データ: `UsageEvent` と `DailySummary` を主データにする。HookイベントからUsageEventへ変換できない場合は対象外として扱う。
- 状態: データなし、同期未実行、未知モデル、料金テーブル古い、予算未設定を明示する。
- 例外: DB接続失敗時はセットアップ手順への導線を出す。

### 候補2: 利用セッション別コスト分析

#### 1. ユーザー課題

- 個人開発者は「どの作業で高くついたのか」を後から振り返りづらい。
- Agent、Composer、Ask、Tabといった機能単位の差は分かっても、実際の開発作業や会話単位の投資対効果が見えにくい。
- チーム管理者は、将来的にプロジェクトや作業カテゴリごとの利用傾向を見たい。

#### 2. 企画する機能

- **機能名**: 利用セッション別コスト分析
- **概要**: `conversationId`、`sessionId`、`workspaceRoot`、`composerMode` を軸に、セッションごとの推定コスト、時間、利用モデル、ツール実行回数を表示する。
- **対象ユーザー**: Primary: 個人開発者 / Secondary: チーム管理者
- **提供価値**: 「高コストだった作業」と「成果につながった使い方」を振り返れる。

#### 3. ユースケース

週末に個人開発者がセッション分析を開き、あるリファクタリング作業でAgentを長時間使い、コンパクションも複数回発生していたことを確認する。次回は事前に設計メモを整理し、Agentに渡す範囲を小さくしてコストを抑える。

#### 4. MVPスコープ

作るもの:
- セッション一覧
- セッション詳細: 期間、モデル、モード、推定トークン、推定コスト、主要イベント
- 高コストセッションのランキング

作らないもの:
- 成果物品質の自動判定
- プロンプト本文の全文分析
- チームメンバー比較

#### 5. 成功指標

- 高コストセッションの閲覧率
- セッション詳細からモデル/機能内訳を見る比率
- ユーザーが作業の改善点を説明できる定性フィードバック

#### 6. 優先度

| 観点 | 評価 | 理由 |
|---|---|---|
| Impact | 中-高 | 具体的な改善行動に結びつきやすい |
| Confidence | 中 | Hooksのsession/conversation情報は使えるが、課金単位との紐付け精度に検証が必要 |
| Effort | 中-高 | 複数テーブル横断とセッション正規化が必要 |

**総合優先度: P1**

#### 7. リスクと対策

- 仕様変更: sessionIdやconversationIdが欠落する場合に備え、timestampとworkspaceRootで補助的にグルーピングする。
- データ精度: 「課金額」ではなく「推定コスト」と明示する。
- セキュリティ: prompt/text/contentはデフォルト非表示、保存済みデータもマスキング方針を検討する。
- UX: セッションが多すぎる場合は高コスト順と期間フィルタを優先する。

#### 8. 開発チームへの要求

- 画面: `/sessions`、`/sessions/[id]`
- API: `GET /api/sessions`、`GET /api/sessions/:id`
- データ: `SessionEvent`、`AgentEvent`、`ToolEvent`、`UsageEvent` を `conversationId` / `sessionId` で集約する。
- 状態: 紐付け不能、トークン情報なし、途中終了、バックグラウンドAgentを区別する。
- 例外: 個人情報を含みうる項目はレスポンスで最小化する。

### 候補3: データ信頼性・プライバシーセンター

#### 1. ユーザー課題

- 非公式APIやHooksに依存するツールでは、ユーザーが「この数字はどれくらい信用できるか」を判断しづらい。
- 個人開発者はローカル利用でも、セッショントークン、プロンプト、ファイル内容が保存されることに不安を持つ。
- Cursor仕様変更や料金体系変更により、突然データが欠損・不正確になる可能性がある。

#### 2. 企画する機能

- **機能名**: データ信頼性・プライバシーセンター
- **概要**: データ収集状態、最新同期、未知イベント、未知モデル、料金テーブル更新日、保存されるデータ種別、削除/マスキング設定を1箇所で確認できる。
- **対象ユーザー**: Primary: 個人開発者 / Secondary: チーム管理者
- **提供価値**: ローカルで安心して使える透明性を提供し、数値の過信・誤解を防ぐ。

#### 3. ユースケース

新規ユーザーがセットアップ後にプライバシーセンターを開き、トークンは保存されないこと、プロンプト本文はコスト計算に不要であること、料金テーブルがいつ更新されたかを確認する。必要に応じてプロンプト保存を無効化し、安心して継続利用する。

#### 4. MVPスコープ

作るもの:
- データ収集ステータス
- 保存データ種別の説明
- 料金テーブル更新日・未知モデル一覧
- プロンプト/ファイル内容の保存有無設定

作らないもの:
- 自動料金テーブル更新
- 公式API移行機能
- 高度な監査ログ

#### 5. 成功指標

- セットアップ後の不安・離脱に関する問い合わせ減少
- 未知モデル・未知イベント発生時の検知率
- プライバシー設定変更後の継続利用率

#### 6. 優先度

| 観点 | 評価 | 理由 |
|---|---|---|
| Impact | 中 | 信頼性と安心感に効くが、直接のコスト可視化価値は候補1より低い |
| Confidence | 高 | 現在のHook保存基盤から状態表示しやすい |
| Effort | 中 | 設定UI、マスキング、データ保持ポリシーの整理が必要 |

**総合優先度: P1**

#### 7. リスクと対策

- 仕様変更: 未知イベントを保存前に捨てるのではなく、検知ログとして扱う余地を残す。
- データ精度: 「信頼性ステータス」を過度に単純化せず、根拠を併記する。
- セキュリティ: 既存DBに保存済みのセンシティブデータ削除導線が必要。
- UX: 初心者にも分かる説明と、詳細を見たいユーザー向けの展開表示を分ける。

#### 8. 開発チームへの要求

- 画面: `/settings/data-quality` または `/privacy`
- API: `GET /api/health/data-quality`、`GET/PUT /api/config/privacy`
- データ: `Config`、Hook各テーブル、料金テーブルメタデータ、未知モデル検出結果
- 状態: 正常、注意、要対応、未設定
- 例外: マスキング設定変更後に既存データへ適用するかを明示する。

# 最優先PRD: 月次コスト・予算ダッシュボード

## 1. ユーザー課題

Primaryユーザーである Cursor Pro/Pro+/Ultra の個人開発者は、AI支援を日常的に使うほど「便利さ」と「コスト不安」の両方を抱える。特に以下の場面で困る。

- 月の途中で、現在の利用ペースがプラン上限内か分からない。
- 高コストの原因がモデル、機能、セッション、特定日のどれなのか分からない。
- Cursorの表示とローカル推定値の差分やデータ鮮度が分からず、数字をどこまで信じてよいか迷う。
- コストを下げたいが、次に取るべき行動が分からない。

Secondaryユーザーであるチーム管理者にとっても、将来的なチーム利用の前段として、個人単位で説明可能なコスト集計モデルが必要になる。

## 2. 企画する機能

- **機能名**: 月次コスト・予算ダッシュボード
- **概要**: 当月の推定消費額、プラン上限、残予算、月末着地予測、モデル別・機能別のコスト要因、データ信頼性をトップページにまとめる。
- **対象ユーザー**: Cursor Pro/Pro+/Ultra の個人開発者をPrimary、チーム管理者をSecondaryとする。
- **提供価値**:
  - 月末の想定外コストを早期に察知できる。
  - 高コスト要因を見つけ、モデル選択やAgent利用方法を見直せる。
  - 推定値の根拠と限界を理解したうえで安心して使える。

## 3. ユースケース

### ユースケースA: 月中の使いすぎに気づく

1. ユーザーが朝に `cursor-consumption` のトップページを開く。
2. 当月の推定消費額がプラン上限の75%に達している。
3. 月末予測がプラン上限を20%超える見込みとして表示される。
4. ユーザーはモデル別内訳で高価なモデルのAgent利用が多いことを確認する。
5. 以後、軽い質問は低コストモデル、長い自動実装は必要時だけ高性能モデルに切り替える。

### ユースケースB: コスト増の理由を説明する

1. ユーザーが前週より消費額が増えていることに気づく。
2. 日別グラフで特定日にスパイクがある。
3. 機能別内訳でAgent利用が増え、モデル別内訳で高単価モデルに偏っていることを確認する。
4. ユーザーは「大規模リファクタリングでAgentを長時間使ったため」と説明できる。

### ユースケースC: 数字の信頼性を確認する

1. ユーザーがダッシュボード上の「推定」ラベルを確認する。
2. chargedCents由来のデータと料金テーブル計算由来のデータ比率を確認する。
3. 料金テーブル更新日と未知モデル有無を確認し、推定値の限界を理解する。

## 4. MVPスコープ

### 作るもの

#### 画面

- トップページ `/`
  - 当月推定消費額
  - プラン上限またはカスタム予算
  - 残予算
  - 消費率プログレスバー
  - 月末着地予測
  - モデル別コスト上位
  - 機能別コスト上位
  - 直近7日の日別消費推移
  - データ鮮度、推定精度、料金テーブル更新状態

- 設定導線
  - プラン: Pro / Pro+ / Ultra / Custom
  - 予算額
  - 請求サイクル開始日

#### API

- `GET /api/usage/summary`
  - 当月消費、予算、残予算、月末予測、消費率、データ鮮度を返す。
- `GET /api/usage/models?range=current_month`
  - モデル別の推定コスト、トークン数、リクエスト数を返す。
- `GET /api/usage/features?range=current_month`
  - 機能別の推定コスト、トークン数、リクエスト数を返す。
- `GET /api/usage/daily?days=7`
  - 日別推移を返す。
- `GET/PUT /api/config/budget`
  - プラン、予算額、請求サイクル開始日を取得・保存する。

#### データ

- 主データ:
  - `UsageEvent`
  - `DailySummary`
  - `Config`
- 補助データ:
  - `AgentEvent.inputTokens/outputTokens/cacheReadTokens/cacheWriteTokens`
  - Hookイベントの `model`、`composerMode`、`conversationId`
- 料金:
  - `pricing/models.json` または同等の外部化された料金テーブル

#### 状態表示

- データなし
- 予算未設定
- 同期未実行
- 最終データ更新が古い
- 未知モデルがある
- chargedCentsがないため推定計算している
- 料金テーブルが古い、または更新日不明

### 作らないもの

- デスクトップ通知やSlack通知
- 自動コスト削減提案の詳細レコメンド
- チームメンバー別請求分析
- Enterprise Admin API連携
- CSV/JSONエクスポート
- Cursor内部APIへの新規同期機構
- 公式課金額との完全照合

## 5. 成功指標

### 定量指標

- 初回セットアップ後にトップページで当月消費額が表示されるユーザー比率
- ダッシュボード閲覧後にモデル別/機能別内訳へ進む比率
- 予算設定完了率
- 未知モデルまたは料金テーブル不備がある場合に、ユーザーへ警告表示できた割合
- 月末予測が表示可能なユーザー比率

### 定性指標

- ユーザーが「今月の消費状況」を説明できる。
- ユーザーが「どの利用を見直すか」を1つ以上挙げられる。
- 推定値の根拠と限界が理解でき、数字への不信や過信が減る。

## 6. 優先度

| 観点 | 評価 | 判断 |
|---|---|---|
| Impact | 高 | プロダクトの中心価値であるコスト可視化に直接効く。ユーザーが次の行動を取りやすい。 |
| Confidence | 中 | 既存のUsageEvent/CostCalculatorで実現可能。ただしHook由来データだけでは公式請求額と差分が出る可能性がある。 |
| Effort | 中 | 既存DB・Prisma・Next.js構成を活かせる。集計API、設定、UI状態設計が主な作業。 |

**優先度: P0**

この機能を最優先にする理由は、現状の「Hookイベントが保存されている」状態から「ユーザーがコストを理解し、使い方を変えられる」状態へ最短で価値を引き上げるためである。セッション分析やプライバシーセンターも重要だが、まずトップページで月次コスト判断ができなければ、ツールの利用目的が伝わりにくい。

## 7. リスクと対策

### 仕様変更リスク

- **リスク**: Cursor Hooks のpayload、モデル名、トークン項目が変わる。
- **対策**:
  - 未知モデルを0円で黙殺せず、警告として表示する。
  - 保存時は既知フィールドを優先し、未知フィールドは破壊しない方針を維持する。
  - 集計APIでは欠損項目を明示し、部分的な集計として扱う。

### データ精度リスク

- **リスク**: chargedCentsがない場合、料金テーブル計算は公式請求額と一致しない可能性がある。
- **対策**:
  - UIで「公式請求額」ではなく「推定消費額」と表記する。
  - chargedCents由来と推定計算由来の割合を表示する。
  - 料金テーブルの更新日とバージョンを表示する。

### セキュリティ・プライバシーリスク

- **リスク**: prompt、file content、tool input/outputがDBに保存され、ユーザーが不安を感じる。
- **対策**:
  - コストダッシュボードでは本文を使わない。
  - APIレスポンスはコスト計算に必要な集計済み数値のみ返す。
  - 将来的なプライバシーセンターで保存制御・削除導線を提供する。

### 料金体系変更リスク

- **リスク**: Cursorのプラン上限、モデル料金、Auto/Composerプール、Max Mode係数が変わる。
- **対策**:
  - 料金テーブル外部化を前提にし、UIに料金テーブル更新状態を出す。
  - プラン予算は固定値だけでなくカスタム入力を許可する。
  - 未対応料金要素は「含まれていない可能性あり」と明示する。

### UXリスク

- **リスク**: 数字が多く、ユーザーが何を見るべきか分からない。
- **対策**:
  - トップカードは「使った」「残り」「このままの予測」「主な要因」に絞る。
  - 詳細はモデル別/機能別カードへ段階的に開く。
  - 警告は1つの主要アクションに結びつける。

### 運用リスク

- **リスク**: ローカルPostgreSQLやNext.jsが停止しているとデータが欠損する。
- **対策**:
  - 最終データ更新時刻を表示する。
  - Hook audit.json をバックアップとして扱う既存方針を維持する。
  - 欠損期間がある場合は「この期間のデータは未取得」と表示する。

## 8. 開発チームへの要求

### 画面要件

#### `/` 月次コストダッシュボード

表示コンポーネント:
- Summary cards
  - This month estimated spend
  - Budget / plan allowance
  - Remaining budget
  - Projected month-end spend
- Budget progress bar
- Top cost drivers
  - by model
  - by feature
- Daily spend mini chart
- Data quality banner
  - last updated
  - estimated vs charged ratio
  - unknown model count
  - pricing table status

画面状態:
- `empty`: UsageEventがない
- `needsBudget`: 予算未設定
- `staleData`: 最終イベントが一定期間以上古い
- `partialEstimate`: 推定計算が含まれる
- `unknownModels`: 料金テーブルにないモデルがある
- `dbUnavailable`: DB接続に失敗

### API要件

#### `GET /api/usage/summary`

返却例:

```json
{
  "billingPeriod": {
    "start": "2026-04-01",
    "end": "2026-04-30"
  },
  "budgetCents": 2000,
  "spentCents": 1247,
  "remainingCents": 753,
  "usageRate": 0.6235,
  "projectedMonthEndCents": 2340,
  "requestCount": 430,
  "lastEventAt": "2026-04-29T00:00:00.000Z",
  "costBasis": {
    "chargedCentsCount": 120,
    "estimatedCount": 310
  },
  "warnings": ["projected_over_budget", "contains_estimated_costs"]
}
```

#### `GET /api/usage/models?range=current_month`

- モデル別に `costCents`、`requestCount`、`inputTokens`、`outputTokens`、`cacheReadTokens`、`cacheWriteTokens` を返す。
- 未知モデルは `pricingStatus: "unknown"` を付与する。

#### `GET /api/usage/features?range=current_month`

- 機能別に `costCents`、`requestCount`、`tokenTotal` を返す。
- featureは既存方針の `chat`、`agent`、`composer`、`tab`、`ask` を基本にし、判定不能は `unknown` とする。

#### `GET /api/usage/daily?days=7`

- 日別に `date`、`costCents`、`requestCount` を返す。
- 欠損日は0として返し、グラフが途切れないようにする。

#### `GET/PUT /api/config/budget`

保存項目:
- `plan`: `pro` / `pro_plus` / `ultra` / `custom`
- `budgetCents`
- `billingCycleStartDay`
- `currency`: MVPでは `USD` 固定

### データ要件

- `UsageEvent.calculatedCostCents` を主要コスト値とする。
- `UsageEvent.chargedCents` がある場合は、計算時に優先する既存方針を維持する。
- `DailySummary` は日別グラフの高速化に使う。ただし集計欠損時は `UsageEvent` から再計算できる設計にする。
- `Config` に予算設定を保存する。
- 料金テーブルにはモデルごとの単価に加え、将来的に `updatedAt` または `version` を持たせることを検討する。

### 例外ケース

- UsageEventが0件:
  - セットアップ手順、Hooks稼働確認、DB接続確認への導線を表示する。
- 予算未設定:
  - Proを初期候補として提示しつつ、ユーザーに明示設定させる。
- 料金テーブルにないモデル:
  - コスト0として隠さず、未知モデルとして表示する。
- chargedCentsなし:
  - 推定値として扱い、データ品質表示に反映する。
- 請求サイクル日が月末に存在しない:
  - 対象月の最終日に丸める。
- DB接続不可:
  - 500だけで終わらせず、画面で `.env` とPostgreSQL起動確認を案内する。

### 非機能要件

- ダッシュボード初期表示に必要なAPIは、ローカル1年分データで500ms以内を目標にする。
- APIレスポンスにはプロンプト本文、ファイル内容、トランスクリプト本文を含めない。
- 価格計算ロジックと表示ロジックを分離し、料金体系変更時の影響範囲を限定する。
- UI上の金額は小数誤差を避けるため、APIではcents単位、表示でドル表記に変換する。
